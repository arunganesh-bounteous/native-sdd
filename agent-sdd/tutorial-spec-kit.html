Agentic SDD — Spec-Kit Reference

Every Spec File,
Explained in Full

These files are the heart of the system. The agent reads them before touching a single line of code. A vague or incomplete spec file produces vague or incorrect output. This guide shows you exactly what to write, what to avoid, and why each field matters.

How to use this guide

Each section covers one file. Read the section for each file you're about to fill in. Pay particular attention to the Critical fields and Common mistakes — these are the parts developers most often get wrong, and they directly impact agent output quality.

Rule for all spec-kit files: The agent reads these files but never modifies them. They are human-authored, version-controlled, and reviewed in PRs. Treat them like code — concise, accurate, and maintained.

spec-kit/ARCHITECTURE.md

Architecture

Module graph, navigation, threading, networking, storage, known violations, and Architecture Decision Records. The agent reads this on every task.

✦ Tech lead writes Updated when architecture decisions change
Most important file in the system. An ADR that says "we use MVVM because of testability" prevents the agent from introducing MVP patterns. A missing violations list causes the agent to "fix" intentional exceptions. Get this right first.

Fields and sections

SectionImportanceWhat to write
Module Structure tableRequiredOne row per Gradle module — name, language, pattern, DI, one-line note. Helps the agent understand module boundaries.
NavigationRequiredNavGraph type, which Activity/Fragment hosts NavHost, deep link handling pattern.
Threading ModelRequiredEvery async approach in use with the modules that use it. Critical for migration rules to apply correctly.
State ManagementRequiredStateFlow, LiveData, RxJava — where each is used. Agent uses this to know what "new" vs "legacy" looks like.
NetworkingRequiredLibrary + version, base URL strategy, auth mechanism, custom interceptors. Do NOT put this in DATA_MODEL.md.
Known ViolationsCriticalList every class/file that intentionally breaks the pattern and why. Without this, the agent fixes them.
Target StateRequiredWhere the codebase is headed. Agent uses this for new screens and features.
ADRsCriticalOne entry per significant architectural decision. Must include the Reason — not just the Decision.

Writing good ADRs

ADRs are what make architecture knowledge explicit instead of tribal. The agent uses them to understand why — which stops it from "improving" things that are intentional.

❌ ADR without reason — useless to agent

### ADR-001 — Use MVVM
- Date: 2024-01-15
- Decision: We use MVVM pattern.
- Reason: It's a modern pattern.
- Consequence: ViewModels for all screens.

✅ ADR with reason — agent understands the constraint

### ADR-001 — MVVM over MVP for testability
- Date: 2024-01-15
- Decision: Use MVVM with Hilt-injected ViewModels.
  No Presenters. No manual dependency wiring.
- Reason: MVP requires concrete Fragment/Activity
  references in Presenters, making unit tests
  brittle. MVVM with ViewModel + StateFlow allows
  full ViewModel testing with no Android framework.
- Consequence: Agent must never introduce a
  Presenter class. All business logic lives in
  ViewModel or UseCase.

Known Violations — the most commonly skipped section

Every codebase has screens or classes that break the pattern for a legitimate reason. Without this list, the agent treats them as bugs and "fixes" them.

❌ Missing violations list

Agent reads the MVVM rule. Finds LegacyLoginActivity with business logic in onCreate(). Moves logic to a ViewModel — breaking the intent behind a class that's deliberately untouched until the migration ticket.

✅ Violations documented

### Known Architecture Violations

- `LegacyLoginActivity.kt`: Business logic in
  onCreate() — MVP pattern. Intentionally kept
  until TICKET-450 migration. Do not refactor.

- `OrderSummaryFragment.kt`: Calls Retrofit
  directly (no Repository). Tech debt DEBT-003.
  Do not add more direct calls here.

Common mistakes

  • Empty or vague ADR reason "We chose X" without explaining why blocks the agent from understanding what it must protect. Write the reason as a constraint: "Because of Y, the agent must never do Z."
  • Putting base URLs in DATA_MODEL.md DATA_MODEL is for data shapes — not infrastructure config. Base URLs, auth mechanism, and interceptors live here under Networking.
  • Not updating when patterns change If the team moves from LiveData to StateFlow and ARCHITECTURE.md still says "StateFlow for new, LiveData for legacy", the agent will treat new StateFlow code as a violation. Update the Threading Model and State Management sections when the team's approach changes.

Per-ticket checklist

  • Task introduces a new module → add a row to the Module Structure table
  • Task changes navigation approach → update Navigation section
  • Task introduces a new async pattern → update Threading Model
  • A significant architectural decision was made for this task → write an ADR
  • Task works around the pattern intentionally → add to Known Violations
🔵 Rarely updated — most tasks need no change 🟡 Always update when architecture decisions change

spec-kit/CONVENTIONS.md

Conventions

Non-negotiable coding standards with ❌/✅ examples. Naming, patterns, ViewModel rules, DI, Compose, Resources — and the Quality Gate the agent runs on every task.

✦ Tech lead writes Updated when team coding standards change
The Quality Gate section at the bottom is what the agent scans against in Step 6b for every task. It must be accurate for your platform. Android/Kotlin defaults are provided — non-Android teams must replace this table.

Sections and what they control

SectionImportanceWhat to write
Kotlin Core RulesRequiredNon-negotiable language rules with anti-pattern code examples. The agent follows these exactly.
Naming ConventionsRequiredTable of element type → convention → example. Deviating here causes test naming failures in the Quality Gate.
Package StructureRequiredExact folder structure for a feature module. Agent uses this when creating new files.
ViewModel ConventionsCriticalStateFlow, SharedFlow, no Android imports, no direct DataSource calls. Include a full working ViewModel example.
Coroutines and FlowRequiredDispatcher rules, no GlobalScope, no Thread.sleep, stateIn pattern, Result return type.
Compose (if used)OptionalStateless composables, collectAsStateWithLifecycle, Material 3 only, hiltViewModel at screen level only.
ResourcesRequiredstrings.xml, no hardcoded colors/dims. Quality Gate checks this on every file.
DI (Hilt)Required@HiltViewModel, @Singleton scope rules, no companion factories.
Quality GateCriticalThe table the agent applies in Step 6b. Platform-specific. Non-Android teams MUST replace this.

Rules need anti-patterns, not just patterns

❌ Rule without anti-pattern — easy to misapply

## Rules
- Use StateFlow for state.
- No Android imports in ViewModel.
- All coroutines in viewModelScope.

✅ Rule paired with anti-pattern — unambiguous

// ❌ LiveData — banned for new state
private val _user = MutableLiveData<User>()

// ✅ StateFlow only
private val _user = MutableStateFlow<User?>(null)
val user = _user.asStateFlow()

// ❌ Android import in ViewModel — banned
class ProfileViewModel : ViewModel() {
    fun showToast(context: Context) { ... }
}

// ✅ No Android framework references
class ProfileViewModel : ViewModel() {
    private val _message = MutableSharedFlow<String>()
    val message = _message.asSharedFlow()
}

The Quality Gate — platform-specific

This section is at the bottom of CONVENTIONS.md and is the only part of the spec-kit that the agent actively runs as a checklist, not just reads for guidance.

iOS / Swift teams: The setup wizard generates a Swift-specific Quality Gate automatically. Review and extend it — key checks include: no force-unwrap (!), no force-cast (as!), @MainActor for all UI-touching code, no business logic in SwiftUI views, no UIKit imports inside SwiftUI files, structs preferred over classes for value types, and test mocks via protocol conformance (not subclassing).

Web / other platforms: Replace the Quality Gate table entirely with your platform-appropriate checks. The agent reads it from CONVENTIONS.md — not from CLAUDE.md. 
## Quality Gate
> Agent (Step 6b): scan every file you created or modified against every
> check in this table. Fix failures before proceeding.

| Check | Rule |
|-------|------|
| No !! operators | Use ?.let {}, ?: return, or requireNotNull |
| No new LiveData | New state uses StateFlow only |
| No GlobalScope | All coroutines in viewModelScope or lifecycleScope |
| No hardcoded strings | All user-visible text in strings.xml |
| No hardcoded colors/dimensions | Use theme attributes or dimens.xml |
| Sealed types exhaustive | No else on when over sealed class/interface |
| No business logic in UI | Composables/Fragments call ViewModel only |
| ViewModel has no Android imports | No Context, View, FragmentManager |
| DTOs not exposed to UI | Repository returns domain models only |
| Tests follow naming convention | functionName_scenario_expectedResult |

Common mistakes

  • Leaving the Quality Gate as Android defaults for a non-Android project The agent will check for !! operators and strings.xml on a TypeScript project. Replace the Quality Gate table entirely with platform-appropriate checks.
  • Writing rules without code examples "No business logic in UI" is abstract. The agent needs to see what business logic in UI looks like. Every rule needs a ❌ anti-pattern and a ✅ correct version as actual code.
  • Overly long CONVENTIONS.md This file is loaded on every task. A 10-page file wastes tokens on every run. Keep to the most important rules. Platform defaults cover the rest.

Per-ticket checklist

  • Team adopts a new pattern (new library, new Compose APIs) → add a section with ❌/✅ examples
  • Agent Quality Gate check caused a false positive → adjust the rule in the Quality Gate table
  • Platform changes (iOS migration, React Native) → replace Quality Gate table entirely
🟢 Set up once during onboarding 🔵 Updated when team standards change

spec-kit/MIGRATION_RULES.md

Migration Rules

How the agent handles legacy patterns. Loaded on every task. The Scope Guard prevents whole-file rewrites.

✦ Tech lead writes Updated when a new legacy pattern is found
The single most important concept here is the Scope Guard. Without it, the agent applying LiveData→StateFlow rules would rewrite entire ViewModel files when a task only needs one new field added.

The Scope Guard

Every migration rule applies only to lines the agent adds or the immediate surrounding context — never the whole file.

❌ Without Scope Guard — whole file rewrite

Task: "Add isRefreshing state to ProfileViewModel"

Agent reads: "Replace LiveData with StateFlow"

Result: Agent rewrites all 15 LiveData properties in the ViewModel to StateFlow — untested, unreviewed, not part of the task.

✅ With Scope Guard — minimal diff

Task: "Add isRefreshing state to ProfileViewModel"

Agent reads: "Apply only to lines you add"

Result: Existing LiveData untouched. New isRefreshing is StateFlow. Agent logs the existing LiveData as a follow-up recommendation.

💡
The Scope Guard is already in the file template — don't remove it. But also make sure your actual rules don't accidentally override it with language like "replace all X with Y".

Writing a migration pattern entry

Each pattern needs: the rule (what to do and not do), the scope (which files trigger it), and a ❌/✅ code example.

## LiveData → StateFlow

- Do not migrate existing LiveData unless the task explicitly scopes it.
- When touching a ViewModel that uses LiveData:
  - Do not change existing MutableLiveData declarations.
  - Use MutableStateFlow for any NEW state added by this task.
  - Do not mix observe() and collectAsStateWithLifecycle() for the same property.

// ❌ Adding new state as LiveData
private val _isLoading = MutableLiveData<Boolean>()  // banned — new state uses StateFlow

// ✅ New state added by this task uses StateFlow
private val _legacyUser = MutableLiveData<User>()     // existing — do not touch
private val _isLoading = MutableStateFlow(false)       // new state — StateFlow only
// TODO [DEBT-XXX]: migrate existing LiveData to StateFlow

Patterns to document

PatternTrigger conditionMust include
LiveData → StateFlowAny ViewModel touchDon't migrate existing, new uses StateFlow, TODO comment format
RxJava → CoroutinesAny file with RxJava importsDon't add new chains, bridge pattern for interaction
MVP → MVVMAny Presenter touchCreate new ViewModel alongside, don't move existing Presenter logic
SharedPreferences → DataStoreAny prefs read/writeNew keys go to DataStore, don't migrate existing in same task
AsyncTaskAny AsyncTask class touchNever add to existing AsyncTask, new work is suspend fun
Java filesAny .java file touchNew logic in separate Kotlin file, don't convert Java

Common mistakes

  • Not listing a legacy pattern that exists in your codebase If RxJava code exists but there's no rule for it, the agent will either replicate it or try to remove it. Audit your codebase for legacy patterns before filling this file. Every pattern that exists needs a rule.
  • Writing "replace X with Y" without the Scope Guard constraint "Replace LiveData with StateFlow" reads as a global instruction. Always phrase rules as "When touching a file that uses X, do not add new X — new code uses Y."
  • Missing the TODO comment format When the agent leaves old patterns in place, it should mark them. Without a consistent format, debt tracking is inconsistent. Specify the exact comment format: // TODO [DEBT-XXX]: migrate — [reason]

Per-ticket checklist

  • Task touches a legacy pattern not yet in this file → add a new section for it
  • Agent created a TODO comment about a legacy pattern → verify format matches this file
  • Task explicitly migrates a pattern → no change needed (task scope overrides the rule)
🟢 Set up once with all known legacy patterns 🔵 Updated when a new legacy pattern is discovered

spec-kit/MODULE_MAP.md

Module Map

The module registry. Source paths, key classes, patterns, and debt anchors — one entry per module. Android: one entry per Gradle module. iOS: one entry per Swift Package target or logical Xcode group. Used after routing, not for routing.

✦ Tech lead writes Updated when a new module is added
Routing is NOT done here — it is done in context/_index.md. MODULE_MAP is the registry the agent consults after routing to get source paths and key classes. Developers confuse these two — see the comparison below.

A complete module entry

### :app/profile — User Profile & Settings

| Field | Value |
|-------|-------|

Common mistakes

  • Leaving Key classes blank On day 1, before any context files exist, Key classes is the agent's only fallback for finding source files. Always list at least 3–4 key files. Use the files you'd look at first if asked to understand this module.
  • Anchor ID mismatch with TECH_DEBT.md TECH_DEBT.md#profile in MODULE_MAP but TECH_DEBT.md has {#settings}. Use the exact same ID in both files. Convention: lowercase module name with hyphens.
  • Thinking keywords here route the agent Developers add great keywords to MODULE_MAP but forget to copy them to context/_index.md. Add keywords to MODULE_MAP for human readability, then copy to _index.md for actual agent routing.
  • Purpose says only what the module does — not what it doesn't "Profile and settings" — agent might load this for checkout-related tasks. Always include "Does NOT own X" for the nearest related module. Boundaries prevent false routing.

Per-ticket checklist

  • Task creates a new Gradle module → add a new entry at the bottom
  • Task reveals a new important class in an existing module → add to Key classes
  • New entry added → also add a row to context/_index.md and a section to TECH_DEBT.md
🟢 Populated once per module — then stable 🔵 Updated only when a new Gradle module is added

spec-kit/TECH_DEBT.md

Tech Debt Register

Known debt by module with status (OPEN / SCHEDULED / RESOLVED) and precise agent rules per entry. Agent loads only sections for modules the current task touches.

✦ Team writes Updated when debt is confirmed, scheduled, or resolved
The Agent rule field is the most critical part of every debt entry. It must be an exact instruction — not a description of the problem. The agent reads the rule and follows it literally.

Field breakdown

FieldImportanceWhat to write
Section header + anchorCriticalMust match MODULE_MAP.md Known debt link exactly. e.g. ## :app/profile {#profile}
Debt titleRequiredOne-line description: DEBT-001 — [what the debt is]. Auto-increment IDs.
StatusRequiredOPEN / SCHEDULED / RESOLVED. Agent skips RESOLVED entries.
LocationRequiredExact file path(s) — path/to/File.kt:L42 if line-specific. Agent uses this to identify affected code.
ImpactOptionalWhy this is a problem — for human readers. Agent uses Agent rule, not this.
Agent ruleCriticalExact instruction for the agent. Must be actionable. See examples below.
Scheduled ticketOptionalTicket ID if SCHEDULED. Helps humans see what's already tracked.

Writing effective Agent rules

❌ Impact description — not an instruction

| Agent rule | This ViewModel uses LiveData
|            | which makes testing harder.
|            | The code is messy and should
|            | be cleaned up eventually. |

The agent doesn't know what to do or not do. It may try to "clean up" the ViewModel.

✅ Exact instruction — agent knows what to do

| Agent rule | Do not add new LiveData to
|            | ProfileViewModel. New state
|            | added by this task must use
|            | MutableStateFlow. Do not
|            | migrate existing LiveData
|            | in this task. |

Anchor IDs must match MODULE_MAP

❌ Anchor mismatch — link is broken

| Known debt | see TECH_DEBT.md#profile |

Common mistakes

  • Writing impact descriptions instead of agent rules "This causes performance issues" vs "Do not call X from the UI thread — use Y instead." Always write Agent rule as an imperative: "Do not...", "Use X instead of Y", "Always call Z before..."
  • Not adding new debt found during task completion Agent reports new debt in the completion report. Humans review and decide — but it often doesn't get added. Make reviewing "Follow-up recommended" in the completion report a PR review step. Add entries here when confirmed.
  • Anchor ID mismatch with MODULE_MAP The link in MODULE_MAP points to a non-existent section. After adding a new module entry in MODULE_MAP, immediately add its section in TECH_DEBT.md with matching anchor.

Per-ticket checklist

  • Completion report lists new debt → review and add to relevant module section
  • Debt was resolved in this task → move entry to Resolved section with resolution notes
  • New module added → add its section header and anchor immediately (even if "No known debt")
🟡 Review after every task completion report 🔵 Major entries when debt is explicitly tackled

spec-kit/TESTING.md

Testing Standards

Test framework, ViewModel/feature pattern, mocking rules, coverage targets. Android: MockK + Turbine + MainDispatcherRule. iOS: XCTest / Swift Testing + async/await + @MainActor + protocol mocks. Loaded when any task requires tests.

✦ Tech lead writes Updated when framework or approach changes
The naming convention here is enforced by the Quality Gate in CONVENTIONS.md on every task. If tests don't follow functionName_scenario_expectedResult, the agent's self-verification step will flag them — and the agent will rewrite them.

Sections to fill in

SectionImportanceWhat to write
Framework Stack tableCriticalLibrary + version for every test dependency. Agent uses exact library names for imports and patterns.
What to test / not testRequiredLayers that require tests. What's explicitly excluded. Prevents the agent writing tests for Android framework classes.
Naming conventionCriticalEnforced by Quality Gate. Must be exactly functionName_scenario_expectedResult or your team's equivalent.
ViewModel test patternCriticalFull working example with MainDispatcherRule, mockk setup, runTest, Turbine. Agent copies this pattern exactly.
Flow / StateFlow testingRequiredTurbine vs advanceUntilIdle — when to use each. No Thread.sleep. No runBlocking.

Coverage targetsOptional% per layer for new code. Agent uses these to determine when "enough" tests have been written.

Naming convention — the most critical rule

❌ Vague names — Quality Gate fails

@Test fun testLogin()
@Test fun shouldWork()
@Test fun loginTest()
@Test fun verifyEmailValidation()

✅ Three-part names — verifiable, searchable

@Test
fun onLoginClicked_withInvalidEmail_showsEmailError()

@Test
fun onLoginClicked_withValidCredentials_emitsNavigateToHome()

@Test
fun loadUser_whenNetworkFails_returnsErrorState()

@Test
fun updateOptIn_withTrueValue_callsEngageManagerNotRepository()

The ViewModel test pattern — copy exactly

This is the pattern the agent uses for every ViewModel test. Include a real, working example so the agent copies the right structure.

class [Feature]ViewModelTest {

    @get:Rule
    val mainDispatcherRule = MainDispatcherRule()
    // ↑ Required — sets Dispatchers.Main to TestDispatcher

    private val [dependency]: [Interface] = mockk()
    // ↑ Always mock the interface, not the implementation

    private lateinit var viewModel: [Feature]ViewModel

    @Before fun setUp() {
        viewModel = [Feature]ViewModel([dependency])
    }

    // SharedFlow / Channel — use Turbine
    @Test
    fun `[method]_[scenario]_[result]`() = runTest {
        coEvery { [dependency].[method](any()) } returns Result.success([value])
        viewModel.navigationEvent.test {
            viewModel.processIntent([Intent])
            assertThat(awaitItem()).isEqualTo([expected])
            cancelAndIgnoreRemainingEvents()
        }
    }

    // StateFlow — use advanceUntilIdle + .value
    @Test
    fun `[method]_[scenario]_[result]`() = runTest {
        coEvery { [dependency].[method](any()) } returns Result.failure(Exception("error"))
        viewModel.processIntent([Intent])
        advanceUntilIdle()
        assertThat(viewModel.uiState.value.errorMessage).isNotNull()
    }
}

Common mistakes

  • Not specifying Turbine vs advanceUntilIdle — the agent picks one randomly SharedFlow/Channel events need Turbine. StateFlow assertions need advanceUntilIdle. If not specified, the agent mixes them. Explicitly document: "Use Turbine for SharedFlow/Channel. Use advanceUntilIdle() for StateFlow assertions."
  • Mocking data classes private val user: User = mockk() — data classes should be created with real values. Write explicitly: "Never mock data classes. Create them with real test values: User(id = "1", email = "test@test.com")"
  • Missing MainDispatcherRule in the pattern Without this, tests using viewModelScope fail with Module with the Main dispatcher had failed to initialize. Include MainDispatcherRule in the pattern example — it must be the first thing in every ViewModel test class.
  • Leaving coverage targets blank Agent doesn't know when "enough" tests are written. Set targets per layer: ViewModel 80%, UseCase 90%, Repository 70%. These apply to code written in the task only.

Per-ticket checklist

  • Task adds a new framework dependency (e.g. MockWebServer) → add to Framework Stack table
  • Team changes mocking library (Mockito → MockK) → update Mocking rules and examples
  • Team changes coverage targets → update Coverage Expectations table
🟢 Set up once during onboarding 🔵 Updated when framework or approach changes

spec-kit/DATA_MODEL.md

Data Model

Domain entities, API contracts with request/response shapes, and field-level notes. Loaded when a task touches network, DB, or data models.

✦ Data / backend owner writes Updated when new endpoints or domain entities are introduced
The mapping direction is non-negotiable: Network DTO → Domain model → UI model. Never skip a layer. Never expose a DTO to the ViewModel or Fragment. This rule is enforced by the Quality Gate.

Mapping conventions — the most important rule

❌ DTO passed to ViewModel — banned

// Fragment reads raw API response field
binding.switch.isChecked =
    response.data.user.emailMarketingOptIn

// ViewModel exposes DTO
val userData: StateFlow<ApiResponse>

✅ Three-layer mapping — correct

// Layer 1: Repository maps DTO → domain
data class UserPreferences(
    val emailOptIn: Boolean = false,
    val zipcode: String? = null
)

// Layer 2: ViewModel exposes domain model
val preferences: StateFlow<UserPreferences>

// Layer 3: Fragment observes domain model
viewModel.preferences.collect { prefs ->
    binding.switch.isChecked = prefs.emailOptIn
}

Writing an endpoint entry

Each endpoint needs: method + path, who calls it, request JSON, response JSON, field table, and any Agent notes.

#### POST /user/preferences

**Used by:** `[Feature]Service.getUserData()` → `[Feature]Repository.getUserInfo()`

Request:
{
  "access_token": "string",
  "identifiers": { "email": "string" }
}

Response 200:
{
  "success": true,
  "data": {
    "user": {
      "email_marketing_opt_in": true,
      "zipcode": "10001",
      "birth_month": 6,
      "birth_day": 15
    }
  }
}

| Field | Kotlin property | Notes |
|-------|----------------|-------|
| data.user.email_marketing_opt_in | UserPreferences.emailOptIn | Authoritative source — use this, NOT [LegacyApi].optin |
| data.user.zipcode | UserPreferences.zipcode | Nullable — shown in profile edit |
| data.user.birth_month | — | Combined with birth_day to form "MM/DD" |

Mapped in: `[module]/.../network/responses/[Feature]Response.kt`

The "already exists" vs "must be added" distinction

This is the most valuable thing you can document in a field table — it prevents the agent from either duplicating existing fields or skipping fields that need to be added.

Note to writeWhen to use it
Already existsField is in the data class — agent must NOT re-declare it
Must be added — does not exist yetField is needed by the task but missing from the data class
Authoritative source — use this, NOT [X]Two sources exist for the same data; one is correct and one is stale
Not currently used in UI — do not addField exists in API but is out of scope for the current task

Common mistakes

  • Putting base URLs and auth here instead of ARCHITECTURE.md DATA_MODEL is for data shapes. Infrastructure config belongs in ARCHITECTURE.md under Networking. Add a note at the top: "Base URLs and auth mechanism live in ARCHITECTURE.md under Networking."
  • Documenting only the response — not the request body For write endpoints, the agent needs the request body shape to know what fields to include. Always document both request and response for write endpoints. Read-only endpoints can have request only.
  • No Agent note for ambiguous fields Two APIs return optin — which one is authoritative? Without a note, the agent picks whichever it reads first. Add Agent note blocks for any field with a gotcha: wrong source, missing implementation, nullable assumption.
  • Not documenting the Kotlin property name alongside the JSON field name email_marketing_opt_in (JSON) → emailMarketingOptIn (Kotlin) — the agent needs both. Use the field table format: JSON field → Kotlin property → Notes. Always include both names.

Brownfield approach — documenting what already exists

Brownfield starting point is different. In greenfield, you design the data model before writing code. In brownfield, DTOs exist across 40 files you haven't fully read, domain models may not be separate from DTOs, and some mappings are workarounds that are now load-bearing. Don't try to document everything on day 1 — write a stale spec and the agent will trust wrong information.

The strategy: document by blast radius, not by completeness. Capture the models the agent is most likely to touch incorrectly — not all models.

Tier 1 — Document immediately (before any tickets run)

Anti-patterns that already exist — the agent needs to know "don't fix these."

_index.md vs MODULE_MAP.md — the split explained

context/_index.md — routing

  • Lightweight keyword → context file table
  • Read by the agent on every task to decide what to load
  • Contains ONLY keywords and context file paths
  • Must be comprehensive — all possible ways a module appears in tickets
  • Updated when a new module/context file is added

MODULE_MAP.md — registry

  • Rich metadata — path, pattern, DI, key classes, debt anchor
  • Read after routing for source paths and fallback files
  • Contains full module description
  • Keywords field is human reference — not used for routing
  • Updated when a new Gradle module is added

When two modules share keywords

💡
If two modules share a keyword (e.g. both "profile" and "settings" modules care about "opt-in"), add two rows — one for each. The agent loads ALL matching context files for a task.

Common mistakes

  • Adding keywords to MODULE_MAP.md but forgetting _index.md MODULE_MAP keywords are never read for routing. The agent only routes from this file. When you fill in MODULE_MAP keywords, immediately copy them here. The setup wizard does this automatically — use it.
  • Using only feature names, no class names Developers often write ticket descriptions with class names: "Fix issue in ProfileViewModel". Without class names in keywords, this doesn't match. Include 2–4 class names from the module as keywords — especially ViewModel and Fragment names.
  • Not updating when a new context file is created Agent generates context/checkout.md after a task, but no routing row exists. Next task on checkout falls back to source file scanning. Every new context file needs a row here. The agent adds a note in the completion report — act on it.
  • Overly narrow keywords that miss synonyms "opt-in" but not "optin", "email subscription", "marketing preference", "promotional emails". Think about all the words your team and PMs use in Jira/Linear tickets for this module. Include all variants.
