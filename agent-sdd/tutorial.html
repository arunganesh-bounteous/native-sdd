Agentic SDD — Spec-Driven Development

AI-First Development
with an Agentic Protocol

A practical guide to running Claude as an autonomous development agent on your codebase — accurately, safely, and at scale.

01What is Agentic AI Development?

Agentic AI means the AI doesn't just answer your question — it reads context, makes decisions, takes action, verifies its work, and reports back. It behaves like an autonomous team member with a defined protocol, not a chatbot you prompt repeatedly.

In a traditional AI workflow you describe the context every session, remind it of the rules, and re-prompt when it goes wrong. In an agentic workflow, the AI navigates your codebase itself, loads the rules it needs, writes the code, checks its own work, and hands back a structured report.

The key distinction: An agent is accountable to its output. Before it marks a task done, it verifies each acceptance criterion with specific evidence — a file, a function, a test. "I implemented it" is not evidence.

The Agent Loop

Read context
→
Understand task
→
Execute
→
Self-verify
→
Self-correct
→
Report

The loop repeats every task. Each iteration makes the agent smarter — it updates context/ files with what it learned, so the next session starts with better knowledge than the last.

02vs Traditional AI Usage

❌ Traditional AI Usage

  • You paste context and rules every session
  • AI answers what you asked, not what you needed
  • No memory between sessions — starts fresh every time
  • You decide what files to show it
  • You manually verify the output
  • Rules drift — AI forgets conventions after 20 messages
  • No audit trail — who did what, when, why

✅ Agentic AI (this approach)

  • Rules live in spec-kit/ — always loaded, never forgotten
  • Agent navigates the codebase via MODULE_MAP.md
  • Knowledge accumulates in context/ after every task
  • Agent decides what files to read — you don't babysit it
  • Agent verifies its own work before reporting done
  • Conventions are non-negotiable — enforced by protocol
  • Completion report is a structured audit trail per task
DimensionTraditionalAgentic (this skeleton)
Context loadingManual — you paste itAutomatic — tiered loading protocol
Rule enforcementHope-basedSpec-kit + quality gate
Task scopeDrifts with conversationBounded by task MD + Out of Scope section
VerificationYou review everythingAgent self-verifies, then you review the report
Knowledge retentionLost each sessionGrows with every PR via context files
PredictabilityVaries with prompt qualityConsistent — protocol is fixed

03The Mental Model

Three types of files. Each has a clear owner and a clear purpose. The agent never confuses them.

📋
spec-kit/ — "What the project IS"
Architecture decisions, coding standards, data models, known debt. Humans write it. Agent reads it. Changes rarely.
🔄
context/ — "What each module looks like NOW"
Key files, state shapes, gotchas, test coverage. Agent writes it after every task. Grows smarter over time.
✏️
tasks/ — "What to do NEXT"
One MD file per ticket. Developer writes it. Agent executes and reports. Tasks stay as a log after merge.
⚙️
CLAUDE.md — "How to BEHAVE"
The agent's 8-step protocol. Every session starts here. Never edit this unless intentionally changing agent behaviour for all future tasks.
The two-line summary: spec-kit is the project's long-term memory. context is the project's short-term, task-updated memory. Together they let the agent work accurately without being spoon-fed context every time.

05spec-kit/ — Project Memory

The spec-kit is the agent's knowledge base. Every file is human-authored, version-controlled, and loaded automatically before each task. The agent never modifies these files — it only reads them.

ARCHITECTURE.md
Architecture & Decisions
Module structure, threading model, navigation, networking, storage. Critically: ADRs (Architecture Decision Records) — dated decisions with Kotlin code examples, consequences, and the reason behind each choice.
Tech lead writes
CONVENTIONS.md
Coding Standards
Non-negotiable rules with ❌/✅ code examples. ViewModel pattern, UiState shape, Intent sealed class, DI, Compose rules. Every rule the agent must follow exactly.
Tech lead writes
MODULE_MAP.md
Navigation Backbone
One entry per Gradle module. Keywords, paths, key classes, dependencies, context file pointer. The agent matches task keywords against this table to know which context files to load.
Tech lead writes
MIGRATION_RULES.md
Legacy Handling Rules
How to handle LiveData, RxJava, MVP, AsyncTask, SharedPreferences, direct Retrofit calls. Rules with ❌/✅ examples. Applied only to lines the agent touches — never forces a full migration.
Tech lead writes
TECH_DEBT.md
Debt Register
Known debt entries per module with status (OPEN/SCHEDULED/RESOLVED), location, impact, and a precise agent rule. The agent loads only the sections relevant to the current task.
Team writes
TESTING.md
Testing Standards
Framework stack, ViewModel test pattern with MockK + Turbine, mocking rules, test file location convention, coverage targets per layer. Loaded whenever a task requires tests.
Tech lead writes
Important: Each spec-kit file starts with > Agent: ... — a blockquote directive that tells the agent when to load this file and what the non-negotiable rules are. The agent reads this first.

What makes a spec-kit file effective

PatternWhy it works
❌/✅ code examplesThe agent pattern-matches on code, not descriptions. Showing the wrong pattern alongside the right one makes violations obvious.
Pipe tables for structured dataRenders as visual tables in VS Code preview and GitHub. Easier to scan than bullet lists.
"Agent must never" phrasingExplicit negative constraints are more reliably followed than positive-only rules.
ADRs with datesContext for why a decision was made — prevents the agent from "improving" things that were intentional.

06context/ — Living Docs

Context files are the agent's accumulated knowledge about each module. Unlike spec-kit (which is static), these files grow smarter with every task. The skeleton gets more accurate over time.

After every task the agent updates the relevant context/<module>.md with what it learned: new files added, state shapes changed, new debt discovered, gotchas encountered.

What a context file contains

SectionContentsUpdated when
Key FilesTable of the most important source files with one-line descriptionsA new file is created
State ManagementWhat the ViewModel exposes — UiState shape, sealed class definitionsState changes in any task
Known DebtDebt discovered during this task (full detail stays in TECH_DEBT.md)New debt found
TestsTest file → scenarios mapping, coverage gapsNew tests written
What the Agent Should KnowNon-obvious facts, gotchas, invariants, must-not-remove workaroundsAgent discovers something surprising
Review context file changes in PRs. They appear alongside the code changes. If the agent wrote something incorrect, correct it — that correction will persist for all future sessions.

The routing index

The context/_index.md is a keyword routing table. When the agent reads a task MD, it matches task keywords against this index to know which module context files to load. You seed it when adopting the skeleton; the agent adds to it as new modules are discovered.

07tasks/ — Work Items

One MD file per ticket. The developer writes it; the agent executes it. The task MD is the contract — precision in, precision out.

Anatomy of a task MD

# Task: [PROJ]-1234 — Track store_selected Firebase event

## Type
- [x] Feature

## Description
Add a store_selected Firebase Analytics event fired when the user selects a store
in SelectStoreFragment. Firebase only — Adjust must not receive this event.

## Acceptance Criteria
- [ ] store_selected event fires when onStoreSelected() is called in SelectStoreFragment
- [ ] Event includes store_id property with value store.id (String)
- [ ] Event is NOT tracked in AdjustAnalytics

## Quality Gate
- [ ] No !! operators introduced
- [ ] No new LiveData — new state uses StateFlow
- [ ] No hardcoded strings

## Out of Scope
- Do not modify AdjustAnalytics.kt
- Do not add store_selected tracking for delivery flow (separate ticket)

## Affected Areas
analytics, store selection, SelectStoreFragment, Analytics.kt

## Testing
Required: Y
Level: Unit
Scenarios:
- [ ] trackStoreSelected() called with correct store_id when onStoreSelected() fires
- [ ] AdjustAnalytics NOT called during store selection

The five rules for effective task MDs

R1
Acceptance criteria are verifiable statements
Write as: "[thing] happens when [condition]" or "[value] equals [expected] when [condition]". The agent names the specific file and function as evidence. "It works correctly" is not verifiable.
R2
Out of Scope is as important as the description
If you don't say what NOT to do, the agent may implement related things it thinks are helpful. Always list at least one exclusion, even if obvious.
R3
One task = one ticket
Don't bundle backend + UI + analytics. Each is a separate task MD. Bundled tasks produce larger, harder-to-review diffs and increase the risk of something going wrong.
R4
Affected Areas accelerates context loading
Even a one-word hint saves the agent from scanning wrong modules. "analytics, SelectStoreFragment" is enough. Without it the agent still finds the right files — just slower.
R5
Test scenarios are prompts, not just checkboxes
If you list scenarios, the agent writes tests for exactly those. If you leave it blank, the agent writes tests based on what it thinks matters. Both work — listed scenarios produce more targeted tests.

08The 8-Step Agent Protocol

Every Claude session starts by reading CLAUDE.md. This file defines a fixed 8-step protocol the agent follows for every task — no exceptions. It cannot be overridden by the task MD.

S0
Read project.config.md
Extract codebase path, platform, language, default_tests flag, branch convention. All file paths resolve from the codebase_path. This is the foundation for everything that follows.
S1
Load Tiered Context
Six tiers loaded in order: always load ARCHITECTURE + MODULE_MAP, always load CONVENTIONS + MIGRATION_RULES + TESTING, then match keywords to load module context files, then tech debt for legacy files, then DATA_MODEL if network/DB, then source files from context.
S2
Read the Task MD
Extract ticket ID, type, description, acceptance criteria, out-of-scope constraints, affected areas, testing requirements. The acceptance criteria become the definition of done.
S3
State Understanding Before Touching Code
Agent writes out affected modules, context files loaded, files to create/modify, which migration rules apply, which debt is in scope, and any ambiguities — before writing a single line of code. If ambiguous: stop and ask.
S4
Confirm Tests
Uses task MD's Required: Y/N if present. Falls back to project.config.md default_tests flag. Only asks the developer if both are absent and N.
S5
Execute
Write code in order: new files first (models, repositories, use cases), then modify existing files, then test files. Follows CONVENTIONS.md and MIGRATION_RULES.md exactly.
S6
Self-Verification — do not skip ✓
The agentic feedback loop. See next section for detail. Agent verifies criteria with evidence, runs quality gate, logs self-corrections, issues PASS or BLOCKED verdict. Only proceeds if PASS.
S7
Update Context Files
After verification passes: update context/<module>.md for every module touched. Add new files, update state shapes, log new debt, update tests section, add non-obvious discoveries.
S8
Write Completion Report
Structured report: files created/modified, context files updated, tests written, acceptance criteria with evidence, quality gate results, self-corrections, follow-up debt. This is the audit trail.

The Ambiguity Protocol

At Step 3, if any acceptance criterion is unclear or contradictory, the agent:

1. Quotes the exact text that is unclear
2. Asks ONE specific question
3. STOPS — does not write code until the developer responds

Do not assume. Do not interpret charitably and proceed. Stop and ask.

This prevents the most common failure mode: agent makes a plausible-sounding assumption, writes 200 lines, and all of it solves the wrong problem.

09The Self-Verification Loop

This is the core of the Agentic AI approach. After executing, the agent doesn't just report done — it proves it, catches its own mistakes, and only issues a PASS when all criteria are met with evidence.

6c: Self-correct

6d: Verdict
↓
Evidence required
10 checks
Fix before proceeding
PASS or BLOCKED

6a — Acceptance Criteria Check

For every acceptance criterion, the agent names the specific file, function, or test that proves it is satisfied:

## Acceptance Criteria Check

- [x] store_selected event fires when onStoreSelected() is called
      verified: SelectStoreFragment.kt:L84 — Analytics.trackStoreSelected(store.id)

- [x] Event includes store_id property
      verified: Analytics.kt:L203 — trackStoreSelected(storeId: String) passes storeId to logEvent()

- [x] Event NOT tracked in Adjust
      verified: AdjustAnalytics.kt not modified (confirmed by grep)

- [ ] Unit test confirms event fires with correct store_id
      BLOCKED: SelectStoreViewModelTest.kt doesn't exist yet — will create in this task
"I implemented it" is not evidence. The agent must name the file and line number, or the test class and method name. This forces the agent to actually navigate to the code it wrote — catching errors that a surface-level pass would miss.

6b — Quality Gate

The agent applies the Quality Gate table defined in spec-kit/CONVENTIONS.md. That table is platform-specific and authoritative — Android/Kotlin teams use the default set; iOS, Web, and Backend teams replace the table with their own equivalent checks.

The default Android/Kotlin checks cover:

CheckRule
No !! operatorsUse ?.let {}, ?: return, or requireNotNull
No new LiveDataNew state uses StateFlow only
No GlobalScopeAll coroutines in viewModelScope or lifecycleScope
No hardcoded stringsAll user-visible text in strings.xml
No hardcoded colours / dimensionsUse theme attributes or dimens.xml
Sealed types exhaustiveNo else on when over sealed class/interface
No business logic in UI layerComposables, Fragments, Activities call ViewModel only
ViewModel has no Android importsNo Context, View, FragmentManager
DTOs/Entities not exposed to UIRepository/Manager returns domain models only
Tests follow naming conventionfunctionName_scenario_expectedResult

Non-Android teams: replace this table in spec-kit/CONVENTIONS.md — the agent reads it from there, not from CLAUDE.md.

6c — Self-Corrections

Any fix made during verification is logged transparently. This gives the developer full visibility into what the agent caught and corrected:

## Self-Corrections

- SelectStoreFragment.kt:L84 — used store!! instead of store ?: return
  → changed to: val storeId = store?.id ?: return

- AnalyticsViewModel.kt:L12 — imported android.content.Context unnecessarily
  → removed import, passed string directly

6d — Verdict

Verdict: PASS
All criteria verified with evidence. Quality gate clean. 2 self-corrections made and logged.

--- or ---

Verdict: BLOCKED
Criterion 3 cannot be verified: the /auth/refresh endpoint is not yet deployed in staging.
Please confirm the endpoint URL before I proceed.
Why this changes the review dynamic: The completion report now tells you exactly what the agent verified and how. You're no longer reviewing the code blind — you're reviewing against the agent's own evidence. Spot-check the evidence; if it's accurate, the code is almost certainly correct.

10The Completion Report

The completion report is the structured output of every task. It is the audit trail — you can read it weeks later and understand exactly what the agent did, what it verified, and what it chose not to do.

12Day 1 Bootstrap — Seeding Context on an Existing Codebase

The system is designed for greenfield and brownfield projects equally. On a new project, context files grow naturally from your first tickets. On an existing codebase, you can pre-seed context files with a read-only bootstrap run — before any code changes are made.

The Day 1 problem: Zero context files exist. Without them, the agent falls back to scanning 5–8 source files per task — slower and less accurate. A bootstrap run fixes this in minutes per module, with no code changes.

Two valid approaches

Option A — Bootstrap first (recommended for large codebases)

  • Run BOOTSTRAP_TEMPLATE.md as a read-only task for each key module
  • Agent reads source files, extracts structure, generates context/<module>.md
  • No production code is created or modified
  • Then run real feature/bug tickets — agent has full context from the start
  • Best when you have 5+ modules and want consistent quality from ticket 1

Option B — Let context grow naturally

  • Skip bootstrap, run real tickets immediately
  • Agent generates context files after each task (CLAUDE.md Step 7)
  • First 2–3 tasks per module are slightly slower (source file scan fallback)
  • Context quality is higher — built from actual task execution, not cold read
  • Best for small codebases or when you want to start shipping immediately

How to run a bootstrap task

01
Copy BOOTSTRAP_TEMPLATE.md
cp tasks/BOOTSTRAP_TEMPLATE.md tasks/BOOTSTRAP-profile.md — one file per module.
02
Fill in the Module section
Copy the module name, source path, and key classes from spec-kit/MODULE_MAP.md. Takes 2 minutes — everything you need is already in MODULE_MAP.
03
Run it in Claude Code
"Read CLAUDE.md and execute tasks/BOOTSTRAP-profile.md" — the agent reads source files, generates context/profile.md, and adds a routing row to context/_index.md. No code changes.
04
Review the generated context file
Check "What the Agent Should Know" — add anything it missed (deep link preconditions, nullable fields the UI assumes non-null, workarounds that must not be removed). This is the highest-value 5 minutes you'll spend.
05
Repeat for each frequently-touched module
Prioritise the modules your current backlog touches most. You don't need to bootstrap every module — just the ones in the next 5–10 tickets.

Real example: After running a bootstrap task on a profile module, the agent auto-generated context/profile.md, context/managers.md, and context/api-sdk.md. The profile context file flagged stale dead code left from a previous ticket, identified that a manager class had no interface (testability debt), and documented the authoritative read/write rules for a marketing opt-in flow — non-obvious facts that would otherwise trip up every future agent session touching that area.

13Writing Good Tasks

The acceptance criteria pattern

❌ Vague — cannot be verified

  • Analytics works correctly
  • Store selection is tracked
  • UI shows loading state
  • Error handling is improved

✅ Specific — verifiable with evidence

  • store_selected event fires when onStoreSelected() called in SelectStoreFragment
  • Event includes store_id matching store.id from the AppStore object
  • LoginUiState.isLoading = true while loginUseCase is running
  • LoginUiState.generalError is set with server message on 401 response

Common mistakes

MistakeFix
Vague description ("improve analytics")Describe exactly what event, property, trigger, and class
No Out of Scope sectionAlways list at least one exclusion — prevents scope creep
Bundling multiple concernsSplit backend + UI + analytics into separate task MDs
Criteria that can't be verified ("works correctly")Rewrite as observable behaviour with specific class/method
Blank Affected AreasAdd at minimum the feature name and one class name
Testing: N with no explanationOK — agent will note what could be tested in the report

After the agent finishes

  • Check the completion report — every criterion should be ticked with evidence, or explicitly blocked with a reason
  • Review context file changes alongside code — they appear in the same PR; if the agent wrote something wrong, correct it in that PR
  • Review "Follow-up recommended" — add any new debt to TECH_DEBT.md if it warrants an entry
  • Review Self-corrections — each one tells you something the agent caught that you might want to add to the Quality Gate for future tasks

14Do's and Don'ts

✅ Do

  • Keep spec-kit files concise — 1–3 pages each keeps token cost low
  • Review context file changes in every PR — they're part of the diff
  • Write ADRs when architecture decisions change — even small ones
  • Add debt to TECH_DEBT.md immediately when discovered — don't rely on memory
  • Use the Quality Gate in task MDs to add task-specific checks
  • Split large features into multiple task MDs — one per concern
  • Trust BLOCKED verdicts — the agent stops because it genuinely can't verify something

❌ Don't

  • Edit CLAUDE.md unless intentionally changing protocol for all future tasks
  • Edit spec-kit/ files from inside a Claude session — they are human-authored
  • Skip the Affected Areas field — it costs tokens and time
  • Write acceptance criteria you can't verify (no observable behaviour)
  • Ignore the Self-corrections section — it shows you what quality problems the agent caught
  • Let context files go stale — correct wrong entries in the PR before merging
  • Bundle tasks to "save time" — bundled tasks produce unmergeable diffs

15Evolution Roadmap

This skeleton implements Phase 1 of a mature Agentic AI development approach. The phases below represent the natural evolution — each builds on the previous.

Phase 1 — Now
Self-Verification Loop ✓
Agent verifies its own work before reporting done. Acceptance criteria require evidence (file + line). Quality gate catches violations. Self-corrections are logged. PASS/BLOCKED verdict before closing the loop. This is the current implementation.
Phase 2 — Next
Living Context — Agent Accumulates Knowledge
Formalise context file updates: define what "a good update" looks like, add a context review checklist, and establish a policy for when context files expire. The skeleton already supports this — Phase 2 makes it rigorous.
Phase 3 — Near Future
Multi-Agent Roles
Split the single agent into specialised roles: Planner (breaks features into subtasks), Coder (executes one subtask), Reviewer (checks diff against spec-kit rules), Tester (writes and runs tests). Each has its own prompt file in agents/. The Planner spawns the others.
Phase 4 — Future
Observability — Audit Trail and Metrics
A reports/ folder where each completed task drops a structured JSON report: rules applied, violations found, debt logged, files touched, self-corrections made. Over time this becomes a leadership dashboard: agent rule compliance rate, debt growth/reduction, coverage trends.

This knowledge doesn't live in someone's head — it's in version control.
