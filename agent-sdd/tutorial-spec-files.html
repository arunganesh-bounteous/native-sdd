Per-ticket spec files

MODULE_MAP.md &
DATA_MODEL.md

The two spec-kit files you update most often. This guide explains what to write, when to write it, and what happens if you skip it.

01What are these files?

They are the agent's navigation system and API contract reference. Without them the agent can still work — but it wastes time scanning the wrong files and risks guessing field names incorrectly.

FileAnswers the questionAgent uses it to
MODULE_MAP.md Where does each feature live in the codebase? Match task keywords → load the right context/ file → read only the relevant source files
DATA_MODEL.md What do the API requests and responses look like? Know exact field names, types, and which fields already exist vs need to be added

Think of MODULE_MAP as a directory and DATA_MODEL as an API contract reference. Both are append-only registers that grow with your project.

02The Golden Rule

Always append. Never replace. Both files are registers — they grow over time. Add new entries for new modules and new endpoints. Never remove or overwrite existing entries. Future tasks depend on what you wrote today.

❌ Wrong — replacing

  • Delete old module entries and rewrite for the new ticket
  • Replace the whole endpoints section for a new task
  • Start from scratch each sprint

✅ Right — appending

  • Scroll to the bottom of the module list, add new module entry
  • Scroll to the bottom of API Contracts, add new endpoint section
  • After 10 tickets the file is nearly complete — future tasks need no changes

03When to Update Each File

Before every ticket, ask two questions. If the answer is yes — add the missing entry. If no — don't touch the file.

Every ticket
Write the task MD
tasks/[PROJ]-XXXX.md — always required. This is the only file you always create.
Often — first few tickets
Add to MODULE_MAP.md
Only if the task touches a module not yet listed. After your first 10 tickets covering different areas, most new tickets need no MODULE_MAP changes.
Often — new API work
Add to DATA_MODEL.md
Only if the task calls an endpoint not yet documented, or adds/changes a field on an existing request. UI-only tasks rarely need DATA_MODEL changes.
Rare
Update ARCHITECTURE.md, CONVENTIONS.md, etc.
Only when the architecture or team conventions actually change — not per ticket.

Decision guide — MODULE_MAP.md

Add entry
The task touches a module/package not yet listed in MODULE_MAP.md
Add a new entry at the bottom with accurate keywords. The agent uses keywords to route — vague or missing keywords mean it loads the wrong context file or scans everything.
Skip
The module is already listed
No action needed. The agent will find it and load the correct context file.
Update keywords
The module is listed but your task uses terms not in the Keywords field
Add the new terms to the Keywords field of the existing entry. Do not create a duplicate entry.

Decision guide — DATA_MODEL.md

Add endpoint
The task calls an API endpoint not yet documented
Add a full endpoint section — method, path, request JSON, response JSON, field table with notes. Without this the agent may guess field names incorrectly.
Add field note
The endpoint is documented but the task adds a new field that doesn't exist yet
Add a row to the field table for that endpoint with a note like "Must be added — does not exist yet". This tells the agent exactly what to create vs what already exists.
Skip
The endpoint and all fields are already documented
No action needed. The agent reads the existing contract.

04MODULE_MAP.md — Purpose

The module registry. Once the agent has routed to the right context file (via context/_index.md), it comes here for source paths, key classes, and debt anchors. It never scans the full codebase from scratch. Android projects use Gradle module names as entries; iOS projects use Swift Package Manager targets or logical Xcode groups.

The routing chain: Task keywords → context/_index.md keyword match → load context/profile.md → read only files listed there → execute task

MODULE_MAP.md is the registry consulted after routing — for source paths, key classes, and debt anchors. Routing lives in context/_index.md.

If a module has no context file yet, the agent uses the Key classes field in MODULE_MAP.md to find the right source files. That still works — but it's slower and less accurate than having a precise context file.

05MODULE_MAP.md — Field Breakdown

Each module entry is a markdown table with 9 fields. Here's what each field does and which ones matter most.

Module name
Gradle module name or logical grouping name. e.g. :app/profile or :feature-api-sdk. Used as the section header.
Required
Path
Relative path from codebase_path in project.config.md. Agent resolves all file reads from this path. Android: app/src/main/java/…/feature/. iOS: Sources/FeatureName/ or MyApp/Features/Auth/.
Required
Keywords
Human-reference terms for this module — class names, feature names, ticket terminology. Kept in sync with context/_index.md, which is the authoritative routing table the agent actually uses. Write keywords here so humans can read and maintain the map; copy them into _index.md so the agent can route.
Critical
Purpose
One sentence — what this module does and what it does NOT do. Boundaries matter — prevents the agent loading this module for unrelated tasks.
Required
Key classes
The 3–6 most important files. Agent reads these first when the context file doesn't exist yet.
Required
Context file
Path to the context/ file for this module. Agent loads this file before reading any source. If the file doesn't exist yet the agent creates it after the task.
Required
Language
Android: Kotlin / Java / Kotlin+Java. iOS: Swift / Swift+ObjC. Tells the agent which MIGRATION_RULES apply and which syntax to use when creating new files.
Optional
Pattern
MVVM, Infrastructure, Repository, Singleton, etc. Helps the agent pick the right patterns for new code.
Optional
Depends on
Which other modules this imports. Helps the agent understand cross-module impact and load additional context files for multi-module tasks.
Optional

06Writing Good Keywords

Keywords are the most important field. The agent matches the words in your task description against these keywords. If a word in the task doesn't appear in any Keywords field, that module won't be loaded.

What to include

TypeExampleWhy
Feature namesprofile, settings, checkoutTicket descriptions usually mention the feature
Class names[Feature]ViewModel, [Feature]ManagerDevelopers often name classes in ticket descriptions
Domain termsopt-in, optin, marketing, promotionalSame concept, multiple spellings — include all variants
API/endpoint namesgetuser, updateuser, /auth/refreshAPI-related tickets often use endpoint names
User-visible labelsemail subscription, push notificationTickets written from UX perspective use UI labels

❌ Too vague — won't match

  • user, data, settings
  • Generic words that appear in every ticket
  • Only one or two keywords for a large module

✅ Specific — matches reliably

  • opt-in, optin, contact options, promotional, email, marketing, [Settings]Fragment, [Profile]ViewModel
  • Mix of feature terms, class names, and UX vocabulary
  • 8–15 keywords covering all the ways this module might appear in a ticket

This is an example entry for a profile/settings module. Notice how the keywords cover the ticket's terminology from multiple angles — class names, feature names, and ticket language all included.

### :app/profile — User Profile & Settings

08MODULE_MAP.md — Per-Ticket Checklist

  • Read the task MD. Note every module, class, and feature area mentioned.
  • Check each area against the Module Index. Is there a matching entry?
  • If yes — check the Keywords field. Do the task's terms appear? If not, add them to the existing entry.
  • If no — add a new entry at the bottom. Fill in Path, Keywords, Purpose, Key classes, Context file at minimum.
  • Never delete or rewrite existing entries. Only append or update keywords.
Time estimate: 3–5 minutes per new module entry. After your first 8–10 tickets, most new tickets need zero MODULE_MAP changes — the map is already populated.

09DATA_MODEL.md — Purpose

The API contract reference. When the agent needs to call an endpoint or modify a request/response model, it reads DATA_MODEL.md to know the exact field names, types, and which fields already exist in the codebase vs need to be added.

Without DATA_MODEL.md the agent either has to find and read the DTO files itself (slower) or it guesses field names (dangerous — can introduce serialisation bugs). This applies equally to Android (Kotlin Codable / Gson / Moshi) and iOS (Swift Codable structs with CodingKeys).

The real risk of skipping it: The agent might add a field named emailOptIn when the actual JSON field is email_marketing_opt_in. On Android that compiles but fails at runtime (Gson/Moshi skips it). On iOS, a missing CodingKeys entry causes silent decode failure. DATA_MODEL.md prevents this entirely by documenting the exact JSON key → Swift/Kotlin property mapping.

10DATA_MODEL.md — What to Document

SectionDocument whenWhat to include
Mapping Conventions Once at project setup The DTO → domain → UI mapping rule. Anti-patterns. Mapper pattern examples.
Domain Entities When a new domain model is introduced Kotlin data class definition, field/type/source table
Database Schema When a Room entity changes Room version, entity definition, migration strategy
API Contracts → Endpoint When a task calls an endpoint not yet documented Method + path, request JSON, response JSON, field table with notes

The most important part — the field table

Below every request/response JSON block, add a table that maps JSON field names to Kotlin property names and adds notes. This is what saves the agent from guessing.

| Field | Kotlin property | Notes |
|-------|----------------|-------|
| `data.user.email_marketing_opt_in` | `...emailMarketingOptIn` | Authoritative — use this, not Ordering API |
| `fields.email_marketing_opt_in` | `[RequestBody].Fields.emailMarketingOptIn` | Must be added — does not exist yet |
| `fields.zipcode` | `[RequestBody].Fields.zipcode` | Already exists |

The "does not exist yet" and "already exists" notes are critical — they tell the agent exactly what to create and what to leave alone.

11Endpoint Entry Format

Every endpoint section follows the same structure. Copy this pattern for each new endpoint.

#### METHOD /path/to/endpoint

**Used by:** `ServiceClass.methodName()` → `RepositoryClass.methodName()`

Request:
```json
{
  "field_name": "type or example value",
  "nested": {
    "field_name": "type"
  }
}
```

Response 200:
```json
{
  "success": true,
  "data": { ... }
}
```

| Field | Kotlin property | Notes |
|-------|----------------|-------|
| `json.field_name` | `KotlinClass.propertyName` | [what the agent should know about this field] |

Mapped in: `path/to/ResponseClass.kt`

> Agent note: [any non-obvious instruction — e.g., "field X does not exist yet, add it as nullable"]

The Agent Note block

The > Agent note: blockquote at the end of an endpoint section is for non-obvious instructions specific to that endpoint. Use it to tell the agent things the JSON alone can't convey:

Field missing
"email_marketing_opt_in is NOT yet in Fields data class. Add as nullable with null default so existing callers are unaffected."
Use this
Wrong source
"Do NOT use [LegacyApiResponse].optin for marketing opt-in — it reflects only the checkbox on a single transaction."
Use this
Gotcha
"birth_month and birth_day are 0 when not set — treat 0 as null, not a real date."
Use this

12Real Example — DATA_MODEL entries

These are entries written for a marketing opt-in ticket — two endpoints and one domain model. The pattern applies to any API-backed feature.

Domain entity

### UserPreferences (domain model)

data class UserPreferences(
    val emailMarketingOptIn: Boolean = false,
    // sourced from [ApiResponse].data.user.email_marketing_opt_in
    val birthday: String? = null,
    // formatted "MM/DD" from birth_month + birth_day
    val zipcode: String? = null
)

| Field | Type | Source |
|-------|------|--------|
| emailMarketingOptIn | Boolean | [ApiResponse].data.user.email_marketing_opt_in |
| birthday | String? | Formatted from birth_month + birth_day as "MM/DD" |
| zipcode | String? | [ApiResponse].data.user.zipcode |

POST /user/preferences (read path)

#### POST /user/preferences
**Used by:** `[FeatureService].refreshUserData()` → `[FeatureRepository].getUserInfo()`

Request: { "access_token": "string", "identifiers": { "email": "string" } }

Response 200 — key fields:
{ "data": { "user": {
    "email_marketing_opt_in": true,   ← READ THIS
    "birth_month": 6,
    "birth_day": 15,
    "zipcode": "10001"
}}}

| Field | Kotlin property | Notes |
| data.user.email_marketing_opt_in | ...emailMarketingOptIn | Authoritative opt-in — use this, NOT [OtherApi].optin |

POST /user/preferences/update (write path)

#### POST /user/preferences/update
**Used by:** `[FeatureService].updateUserData()` → `[FeatureRepository].updateUserInfo()`

Request fields:
{
  "access_token": "string",
  "identifiers": { "email": "string" },
  "fields": {
    "zipcode": "string",            ← already exists in Fields data class
    "birth_month": 6,               ← already exists
    "birth_day": 15,                ← already exists
    "email_marketing_opt_in": true  ← NOT YET in Fields — must be added
  }
}

| Field | Kotlin property | Notes |
| fields.email_marketing_opt_in | [RequestBody].Fields.emailMarketingOptIn | Must be added — does not exist yet. Add as Boolean? = null |
| fields.zipcode | [RequestBody].Fields.zipcode | Already exists |

> Agent note: Add emailMarketingOptIn as nullable (Boolean? = null) so
> existing callers that don't set it are unaffected.

13DATA_MODEL.md — Per-Ticket Checklist

  • Does the task call any API endpoint? If yes — is that endpoint documented in DATA_MODEL.md?
  • If the endpoint is missing — add a full section: method + path, request JSON, response JSON, field table.
  • Does the task add or change a field on an existing request? Add a row to that endpoint's field table with a "Must be added" note.
  • Is there a non-obvious gotcha about this endpoint? Add an > Agent note: block — wrong source, missing field, edge case values.
  • Does the task introduce a new domain entity? Add it to the Domain Entities section with field/type/source table.
  • Never remove existing endpoint sections — other tickets may reference them.
Time estimate: 5–10 minutes per new endpoint. UI-only tasks and tasks that only use existing, already-documented endpoints need zero DATA_MODEL changes.

14File Lifecycle Over Time

Both files follow the same growth pattern. Heavy at the start, nearly zero-maintenance after the first month.

StageMODULE_MAP.mdDATA_MODEL.md
Ticket 1 (first task) Write 3–5 new module entries for all areas the ticket touches Write 1–3 endpoint sections for all APIs the ticket calls
Tickets 2–5 Add 1–2 new module entries per ticket as new areas are encountered Add 1 new endpoint section per ticket when a new API is called
Tickets 6–10 Mostly no changes — modules already mapped. Occasional keyword additions. Occasional new endpoint or field note. Most endpoints already documented.
Ticket 10+ Almost never changes unless a new Gradle module is added to the project Only changes when a new external API is integrated or a new endpoint is added
The compounding benefit: Every entry you write today saves the next developer (and future tickets) 5–10 minutes of spec prep. After 10 tickets, new developers on the team can write a task MD and run it without any spec-kit changes at all — everything is already documented.

15Common Mistakes

MistakeImpactFix
Replacing instead of appending Future tasks for already-mapped modules fail to load context — agent scans codebase from scratch Always scroll to bottom and add new entries. Never delete existing ones.
Vague keywords (user, data, screen) Agent loads wrong modules or too many modules — slow and inaccurate Include class names, feature names, and ticket terminology. 8–15 keywords per module.
Skipping the field table in DATA_MODEL Agent guesses Kotlin property names from JSON — can introduce serialisation bugs Always add the field table mapping JSON names to Kotlin property names
Not noting "does not exist yet" Agent assumes the field already exists and skips creating it, or creates a duplicate Explicitly annotate each field as "already exists" or "must be added"
Creating duplicate module entries Agent loads both entries, gets conflicting information Search the file before adding. If the module exists, add keywords to the existing entry.
Documenting UI-only tickets in DATA_MODEL Unnecessary work — adds noise Only add to DATA_MODEL if the task touches API contracts or domain entities
